第 4 章 数据管理与可重复分析

野外记录散落 → 设计关系进入数据表 → 可追溯分析数据
① 设计产生字段
样点、年份、重复调查——每一列都来自研究设计中的一项决定
→
② 主键锁定一行
station_id + year + occasion 唯一确定一条记录；宽表/长表各有用途
→
③ 只读原始，追溯清洗
原始数据只读；清洗日志记录每一步；匿名化保护敏感信息
黑豹提醒：数据表是研究设计的可计算表达——设计丢掉的字段，分析永远找不回来。

4.1 数据表必须从研究设计生长出来

第 3 章结束时，你为黄黄草苗圃试验画了一张层级图：区组 → 小区 → 植株 → 叶片。这张图不只是设计工具——它直接决定了你的数据表里每一列叫什么、每一行代表什么、哪些行之间不是独立的。

但很多学生第一次面对真实数据时，打开的是一个”别人给的 Excel 表”。表里有几十列，列名有时是拼音缩写，有时是”备注 1”“备注 2”，有时同一列里既有数字又有文字。更糟的是，你完全不知道这张表是怎么来的——哪些是原始记录、哪些是后来算的、哪些被”顺手改过”。

本章要建立的核心习惯是：在打开任何数据文件之前，先根据研究设计画出数据表的结构。 不是”先看看数据里有什么”，而是”先确认数据应该有什么”。

我们换一个项目来练这件事。从本章开始，红外相机野生动物监测项目（项目 B）将承担数据管理与抽样不确定性的教学任务。你是一名保护区的技术员，负责管理 60 个红外相机样点、连续 3 年的监测数据。

观察问题： 三年下来，你手上有 60 个相机样点的记录——有些样点拍到了目标物种，有些没有；有些相机故障了几个月；有些数据存在巡护员的 U 盘里还没汇总。你怎么把这一切整理成一张能用于分析、也能让后来者看懂的数据表？

认清单位一行代表什么
建立主键唯一标识每条记录
区分状态零、未检出、缺失
记录清洗原始只读，日志可追溯
交付数据字典、质控、匿名化

学完本章后，你能够：

  1. 从研究设计中推导出数据表必须包含的字段和主键。
  2. 区分宽表与长表，并说明一行在不同表型中代表什么。
  3. 区分”真零”“未检出”“未调查”和”缺失”四种状态，并用正确的编码记录。
  4. 写出包含字段定义、单位、合法值和设计层级的数据字典。
  5. 建立原始数据只读、清洗日志可追溯、分析数据可复现的工作流程。
  6. 识别并避免常见的数据管理错误——删除零值、用空白表示缺失、把文件名当作唯一来源。

从一个真实问题开始

保护区的红外相机监测网络覆盖了 60 个样点，分布在核心区、缓冲区和边缘区三种生境类型中。每年每个样点进行 6 次调查（occasion），每次调查记录相机工作的夜数（effort_nights）、是否检出目标物种（detected）、以及检出的独立照片数（photo_count）。

三年下来，理论上应该有 60 × 3 × 6 = 1080 条记录。但实际上，有些相机的电池在雨季耗尽，有些样点的数据还在巡护员的野外记录本上没录入，有些照片因为镜头起雾无法判读。

你现在要做的事，不是立刻跑统计模型，而是先把这些数据整理成一张”忠实保存了研究设计”的数据表。

4.2 用唯一标识保存样点、时间与重复调查

数据表的第一条铁律：每一行必须能被唯一识别。 不能出现”这两行看起来一样，我不知道它们是不是同一条记录的两次录入”。

在红外相机项目中，唯一识别一条记录需要三个字段的组合：

  • station_id：哪个样点（S001–S060）
  • year：哪一年（1, 2, 3）
  • occasion：该年的第几次调查（1–6）

这三个字段合在一起，构成了这张表的复合主键（composite primary key）。任何两条记录在这三个字段上不能完全相同——如果相同，说明数据有重复录入。

注意，station_id 本身不是主键——一个样点有 3 年 × 6 次 = 18 条记录。station_id + year 也不是——每年有 6 次调查。只有三个字段一起，才能唯一锁定”S003 样点在第 2 年第 4 次调查”这一条记录。

为什么这件事重要？因为主键直接对应研究设计中的独立抽样单位 × 重复结构。如果主键设错了，后面所有分析都会在错误的层级上数样本量。

# 读取红外相机数据
camera <- read.csv("../data/teaching/camera_monitoring_long.csv",
                   stringsAsFactors = FALSE)

# 检查总行数
nrow(camera)  # 应为 1080

# 检查主键是否唯一
any(duplicated(camera[, c("station_id", "year", "occasion")]))  # 应为 FALSE

# 检查每个样点每年是否有 6 次调查
table(camera$station_id, camera$year)  # 每格应为 6

关键判断： 主键不是”随便找一列不重复的编号”——它必须对应研究设计中独立单位与重复结构的组合。

4.3 一行代表什么：宽表、长表与分析单位

同一份数据可以用两种形状存储，但它们的”一行”代表完全不同的东西。

长表（long format）： 每行是一个观测。红外相机数据就是长表——一行 = 一个样点 × 一次调查。1080 行，每行有 station_id、year、occasion 和该次调查的测量值。

宽表（wide format）： 每行是一个独立单位，多次观测展开为多列。如果把相机数据转成宽表，一行 = 一个样点，18 列分别对应 3 年 × 6 次调查的检出状态。

两种表型各有用途： - 长表是分析和存储的标准格式——绝大多数 R 建模函数期望长表输入。 - 宽表便于人类阅读和手工检查——你可以一眼看到一个样点三年间的检出模式。

但有一个致命的误解：把宽表的一行当作一个独立样本。 宽表的一行（一个样点）确实是一个独立抽样单位，但它的 18 个观测值之间不是独立的——同一样点的不同年份和不同 occasion 之间存在相关。如果你把宽表的 18 列拆成 18 个”独立”数据点，你就在制造伪重复。

# 演示：宽表 vs 长表的行含义
# 长表：一行 = 一次调查
head(camera[, c("station_id", "year", "occasion", "detected")], 8)

# 转为宽表（检出状态）
camera_wide <- reshape(
  camera[, c("station_id", "year", "occasion", "detected")],
  idvar = "station_id",
  timevar = "occasion",
  direction = "wide"
)
# 宽表：一行 = 一个样点，列 = 不同 occasion 的检出状态
head(camera_wide[, 1:7])

关键判断： 长表的一行是”一次观测”，宽表的一行是”一个独立单位”。两种表型的行数不同，不是因为数据变了，而是因为”一行代表什么”变了。

4.4 零、未检出、未调查和缺失不是一回事

这是本章最容易出错、也最容易在分析中造成系统性偏差的知识点。

在红外相机数据中，detected = 0 有且只有一种含义：这次调查完成了，相机正常工作了足够的夜数，调查员检查了全部照片，没有发现目标物种。 它不等于”目标物种不存在”——可能物种存在但没经过相机，或者经过了但没被拍到。这是第 19 章要解决的问题。

但 detected 字段为空（缺失，NA）可能意味着完全不同的事情：

情况 detected 的值 effort_nights 的值 含义
完成调查，检出 1 正常（5–14） 拍到目标物种
完成调查，未检出 0 正常（5–14） 没拍到，但调查完成了
相机故障 NA 0 或很小 相机没工作，无法判断
数据未录入 NA 正常 调查做了，但数据还没进系统
样点被撤销 NA NA 这个 occasion 根本不存在

如果把后三种情况都留成空白，或者更糟——把空白全部替换为 0——会发生什么？

反例： 某位分析者拿到相机数据后，发现 detected 列有一些空白单元格。他假设”空白就是没拍到”，把所有空白替换为 0，然后计算”检出率 = 检出次数 / 总调查次数”。结果：检出率被人为压低——因为相机故障的月份被当成了”调查了但没拍到”。如果他用这个被压低的检出率去拟合生境模型，会得出”目标物种与生境无关”的错误结论——实际上只是因为故障样点恰好集中在某种生境类型中。

# 区分四种状态
# 完成调查且检出
sum(camera$detected == 1, na.rm = TRUE)

# 完成调查但未检出
sum(camera$detected == 0, na.rm = TRUE)

# 注意：本教学数据集中没有缺失值，所有 1080 行都是完整调查
# 但在真实数据中，你需要分别统计：
# - 缺失的 detected（NA）
# - effort_nights 异常低的行（可能相机故障）
table(camera$effort_nights)
黑豹提醒：空白不等于零

在数据表中，“没有数字”和”数字是 0”是两个完全不同的信息。用空白同时表示”未检出”“相机故障”“还没录入”，等于把三种不同的数据生成过程压缩成一个无法区分的符号。一旦压缩，分析永远无法恢复。

4.5 数据字典与录入规则

数据字典（data dictionary）是一份独立于数据文件本身的文档，它回答：每一列叫什么、是什么意思、取什么值、在设计的哪一层。没有数据字典的数据表，六个月后连你自己都看不懂。

数据字典至少包含以下字段：

字段 说明
变量名 在数据文件中的列名
中文含义 用一句话说清这个变量测量的是什么
数据类型 整数、数值、字符、0/1……
单位或水平 cm、%、“core/buffer/edge”……
设计层级 样点层 / occasion 层 / 样点×年份层……
合法值范围 哪些值可能出现，哪些值如果出现就是录入错误
缺失是否允许 什么情况下允许 NA

本章使用的红外相机数据字典见 data/teaching/DATA_DICTIONARY.md 第 2 节。在开始分析之前，请先完整阅读这份字典。

录入规则同样重要——它们应该在数据开始录入之前就写下来：

  1. 日期格式统一。 不要出现”2025-03-01”“2025/3/1”“3 月 1 日”三种写法。
  2. 分类变量用固定编码。 stratum 只允许 core、buffer、edge 三个值——不允许 Core、核心区、1 等变体。
  3. 数值变量的精度统一。 forest_cover 保留一位小数，不要有些行是 65、有些行是 65.0、有些行是 65.00。
  4. 缺失值统一用 NA。 不要用 -999、缺失、N/A、空白等不同方式表示缺失。

4.6 从原始数据到分析数据的可追溯流程

数据管理的黄金法则：原始数据只读。 任何修改——纠正录入错误、统一编码、计算衍生变量、排除异常值——都必须在一个独立的清洗脚本中完成，并且脚本本身要被版本管理。

三层数据架构：

原始数据层（raw/）         ← 只读，永不做任何修改
    ↓ 清洗脚本（scripts/clean_*.R）
清洗数据层（data/clean/）  ← 可复现生成，记录了每一步修改
    ↓ 分析脚本（chapters/*.qmd）
分析输出（结果、图表）     ← 从清洗数据生成，不直接读原始数据

为什么需要这三层？因为六个月后，当审稿人问你”这个异常值是怎么处理的”，你不能靠记忆回答——你必须能指向清洗脚本的第 27 行，那里记录了”SPAD > 80 标记为可疑，保留但不在主要分析中使用”。

# 演示：生成样点级汇总，但保留原始长表
# 注意：汇总表用于快速查看，不替代原始数据

# 每个样点的总调查次数和检出次数
station_summary <- aggregate(
  cbind(n_surveys = detected) ~ station_id,
  data = camera,
  FUN = function(x) c(total = length(x), detected = sum(x == 1))
)
# 展开列表列
station_summary$total_surveys <- station_summary$n_surveys[, "total"]
station_summary$n_detected <- station_summary$n_surveys[, "detected"]
station_summary$n_surveys <- NULL

head(station_summary)

# 朴素检出率（注意：这不是占域率！见第 19 章）
station_summary$naive_occ <- station_summary$n_detected / station_summary$total_surveys
summary(station_summary$naive_occ)

关键判断： 原始数据只读，清洗脚本可复现，分析数据可追溯——这三条不是”最佳实践”，是数据管理能够被信任的最低条件。

4.7 质控、匿名化与可重复交付

数据管理的最后一步是把数据交付给分析者（可能是六个月后的你自己，也可能是合作者）。交付包至少包含：

  1. 数据文件（CSV 格式，UTF-8 编码，不加密、不压缩为专有格式）
  2. 数据字典（每个字段的定义、单位、合法值）
  3. 清洗脚本（从原始数据到清洗数据的完整代码，含随机种子）
  4. 质控报告（多少行、多少缺失、多少异常值标记、哪些样点数据不完整）
  5. 匿名化说明（哪些字段被脱敏、如何脱敏、脱敏后是否仍可追溯）

对于红外相机数据，匿名化至少涉及： - 坐标脱敏： 如果数据要公开发布，x_km 和 y_km 可以保留（已经是相对坐标），但不应包含经纬度或可反向定位的地名。 - 敏感物种： 如果拍到了极度濒危物种，其具体样点位置不应出现在公开数据中——可以用”样点编号 + 物种存在/不存在”替代。 - 人员信息： 调查员姓名不应出现在数据表中——用调查员编号替代。

反例（续 4.4）： 某分析者把所有 detected 为空的记录直接删除，然后报告”本监测网络三年共完成 1020 次有效调查”。但实际上有 60 次调查是相机故障——删除它们等于假装这些样点在这段时间不存在。如果故障样点恰好集中在边缘区（因为边缘区湿度大、电池更容易耗尽），那么”有效调查”的样点就不再能代表整个保护区的样点构成。抽样框被悄悄地改变了，而分析者自己都不知道。

# 质控：按样点检查数据完整性
# 每个样点应有 18 条记录（3 年 × 6 次）
obs_per_station <- table(camera$station_id)
all(obs_per_station == 18)  # 应为 TRUE

# 检查 effort_nights 的分布
summary(camera$effort_nights)

# 标记努力量异常低的 occasion（< 3 夜可能表示相机故障）
camera$low_effort <- camera$effort_nights < 3
table(camera$low_effort)

# 检查 detected 和 photo_count 的一致性
# detected=0 时 photo_count 应为 0
all(camera$photo_count[camera$detected == 0] == 0)
# detected=1 时 photo_count 应 > 0
all(camera$photo_count[camera$detected == 1] > 0)
黑豹提醒：数据交付不是”把文件发过去”

可重复的数据交付意味着：接收方拿到你的文件后，只运行你提供的脚本，就能从原始数据生成与你完全相同的清洗数据和分析结果。如果中间有一步需要你口头解释，那这一步就是不可重复的。

本章小结

  • 数据表是研究设计的可计算表达。 设计中的每个层级（样点、年份、重复调查）都应对应数据表中的字段；丢失的字段意味着丢失的设计信息。
  • 主键锁定”一行代表什么”。 红外相机数据的主键是 station_id + year + occasion——三个字段共同唯一确定一条记录。
  • 宽表和长表的行含义不同，但数据相同。 长表是分析标准格式；宽表便于人类阅读，但不能把宽表的多列当作独立观测。
  • 零、未检出、未调查和缺失是四种不同的信息。 用空白或 0 统一表示它们，等于在分析之前就引入了无法恢复的偏差。
  • 原始数据只读，清洗可追溯，分析可复现。 三层架构和清洗日志是数据管理能够被信任的最低条件。

练习

  1. 识别层。 以下数据表中，每一行代表什么？主键应该由哪些字段组成？
